# US's

## Persona: Professor

### Funcionalidade 1: Organizar turmas

#### História 1.1: 

Eu, como **professor** posso **editar todos os dados das turmas** para **centralização dos dados**.

#### Critérios de aceitação:

1) O sistema permite que o professor edite todos os dados das turmas de forma abrangente e sem restrições desnecessárias.

2) As alterações realizadas pelo professor nos dados das turmas são refletidas corretamente e de forma consistente em todos os sistemas e interfaces relacionados. 

#### História 1.2: 

Eu, como **professor** posso **criar grupos aleatórios de usuários cadastrados para o início da fase de seleção de grupos** para **centralização dos dados**.

#### Critérios de aceitação:

1) O sistema permite que o professor crie grupos aleatórios de usuários cadastrados de forma eficiente.

2) Os grupos gerados pelo sistema são equilibrados em termos de número de usuários para a seleção de grupos.

3) O sistema fornece ao professor a opção de revisar e ajustar os grupos gerados aleatoriamente, se necessário.

#### História 1.3: 

Eu, como **professor** posso **exportar um arquivo .xls com matrícula, nome, email, equipe e o saldo de PAX de todos os alunos da turma selecionada** para **feedback rápido do desempenho**.

#### Critérios de aceitação:

1) O sistema permite que o professor exporte um arquivo .xls contendo as informações de matrícula, nome, e-mail, equipe e saldo de PAX de todos os alunos da turma selecionada. 

2) O arquivo .xls exportado é gerado de forma precisa e estruturada, garantindo a integridade e a correta formatação dos dados.

#### História 1.4: 

Eu, como **professor** posso **arquivar turmas** para **possuir o histórico das turmas**.

#### Critérios de aceitação:

1) O sistema permite que o professor arquive turmas de forma fácil e intuitiva, sem restrições desnecessárias.

2) As turmas arquivadas são armazenadas e mantidas no histórico do sistema de forma organizada e acessível.

#### História 1.5: 

Eu, como **professor** posso **importar um arquivo .xls com matrícula, nome e email de todos os alunos da turma selecionada** para **centralização dos dados**.

#### Critérios de aceitação:

1) O sistema permite que o professor importe um arquivo .xls contendo as informações de matrícula, nome e e-mail de todos os alunos da turma selecionada.

2) O processo de importação do arquivo .xls é eficiente e confiável, permitindo ao professor centralizar rapidamente os dados dos alunos selecionados.

### Funcionalidade 2: Publicar atividade

#### História 2.1:

Eu, como **professor** posso **definir a sequência das atividades e disciplinas** para **disponibilizar conteúdo**.

#### Critérios de aceitação:

1) O professor deve poder realizar essa tarefa sem dificuldades técnicas significativas, garantindo uma experiência de usuário fluida.

2) O conteúdo disponibilizado segue a sequência estabelecida pelo professor de forma consistente e acessível para os alunos.

#### História 2.2: 

Eu, como **professor** posso **definir o valor em PAX** para **bonificação clara para trazer motivação**.

#### Critérios de aceitação:

1) Os alunos compreendem claramente o valor em PAX definido como bonificação e como ele afeta sua motivação.

2) O sistema armazena e exibe o valor correto, permitindo que tanto o professor quanto os alunos tenham visibilidade em relação aos PAXs definidos.

#### História 2.3: 

Eu, como **professor** posso **definir datas** para **gerenciar atividades realizadas**.

#### Critérios de aceitação:

1) O aplicativo permite o professor inserir as datas de início e término das atividades de maneira intuitiva, sem ambiguidade ou erros de formato.

2) O aplicativo permite o professor definir a data do fim do semestre.

3) O aplicativo permite o professor defirnir as datas de entrega importante.

## Persona: Estudante

### Funcionalidade 3: Organizar equipes

#### História 3.1: 

Eu, como **estudante** posso **jogar jogos durante a dase de seleção de grupos** para **um processo fluido, perda de manos tempo de aula**.

#### Critérios de aceitação:

1) Os jogos disponíveis durante a fase de seleção de grupos são integrados de forma transparente ao processo de seleção, permitindo que os estudantes participem dos jogos sem interrupções.

2) O tempo dedicado aos jogos durante a fase de seleção de grupos é equilibrado e não causa uma perda excessiva de tempo de aula.

#### História 3.2: 

Eu, como **estudante** posso **selecionar participanres para formar minha equipe** para **alterar a equipe de acordo com a dinâmica da aula**.

#### Critérios de aceitação:

1) O sistema deve Validar restrições na seleção de participantes.

2) O sistema deve permitir o aluno Verificar a lista de participantes da sua equipe.

### Funcionalidade 4: Visualizar Equipes

#### História 4.1: 

Eu, como **estudante** posso **visualizar infomações dos integantes de cada grupo** para **centralização das informações**.

#### Critérios de aceitação:

1) O sistema permite que os alunos vizualizem informações dos integrantes de um grupo.

2) O sistema permite validar restrições de acesso às informações do grupo

#### História 4.2: 

Eu, como **estudante** posso **editar infomações do usuário** para **garantir dados corretos e atualizados**.

#### Critérios de aceitação:

1) O sistema permite o estudante editar as informções do usuário.

2) O sistema fornece mecanismos de segurança e validação para garantir que as alterações nas informações do usuário sejam legítimas e autorizadas.

#### História 4.3: 

Eu, como **estudante** posso **visualizar informações do grupo** para **fácil comunicação com toda a equipe**.

#### Critérios de aceitação:

1) O sistema exibe as informações do grupo de forma clara e organizada, facilitando a visualização e compreensão para o estudante. 

2) As informações do grupo são atualizadas em tempo real, refletindo com precisão a composição e os detalhes da equipe.

3) O sistema permite verificar a visualização das informações do grupo.

#### História 4.4: 

Eu, como **estudante** posso **visualizar o ranking de PAX da minha turma** para **observar o progresso de todas as equipes**.

#### Critérios de aceitação:

1) O sistema exibe o ranking de PAX da turma de forma clara e acessível para o estudante. 

2) O ranking de PAX é atualizado regularmente, refletindo com precisão o progresso das equipes ao longo do tempo.

### Funcionalidade 5: Visualizar Atividades

#### História 5.1:

Eu, como **estudante** posso **visualizar as unidades e atividades** para **centralização das informações**.

#### Critérios de aceitação:

1) O sistema exibe as unidades e atividades de forma clara e organizada, facilitando a visualização e navegação para o estudante.

2) As unidades e atividades são atualizadas e refletidas corretamente no sistema.

3) O sistema fornece detalhes relevantes sobre as unidades e atividades, como objetivos, instruções ou datas importantes.

#### História 5.2: 

Eu, como **estudante** posso **visualizar as regras para o cumprimento das atividades** para **clareza nas propostas das atividades**.

#### Critérios de aceitação:

1) O sistema exibe as regras para o cumprimento das atividades de forma clara e acessível para o estudante.

2) As regras são atualizadas sempre que necessário, refletindo as atualizações ou alterações feitas no sistema.

#### História 5.3: 

Eu, como **estudante** posso **visualizar valor dos PAX's de cada atividade** para **pontuação clara**.

#### Critérios de aceitação:

1) O sistema permite o aluno Visualizar valor dos PAX de uma atividade específica.

2) O sistema permite Validar restrições de acesso ao valor dos PAX da atividade.

3) O sistema permite verificar visualização do valor dos PAX das atividades como professor.
